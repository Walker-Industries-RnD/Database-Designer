Best Practices for Database Design

Databases are like towers; follow these tips to ensure you have a good foundation and don't end up needing to remake everything.

1. Follow Normalization Principles

  • Aim for at least 3NF (Third Normal Form) unless you have a specific reason to denormalize (e.g., read performance in reporting).

    PLEASE READ 9. Understanding 3NF

  • Eliminate redundant data by splitting related information into separate tables.

  • Use reference columns (foreign keys) instead of duplicating values.

  • Tip: The designer’s reference validation on build will catch broken or dangling references early.

2. Choose Meaningful and Consistent Naming

  • Use singular, PascalCase or snake_case table names consistently (e.g., UserProfile, user_profile).
  • Column names should be clear and descriptive (e.g., EmailAddress instead of email or str).
  • Prefix related tables with a common schema (e.g., Auth.User, Auth.Role, Auth.Permission).
  • Avoid reserved keywords (the build process auto-escapes C# keywords, but better to avoid them upfront).

3. Define Primary Keys Properly

  • Every table should have a single, stable primary key.
  • Prefer surrogate keys (e.g., Id BIGINT with SERIAL or IDENTITY) over natural keys for simplicity and performance.
  • Mark the primary key column with Is Primary checked.
  • Use Is Unique + Is Not Null for natural keys when needed.

4. Use Constraints Liberally

  • Enforce data integrity at the database level:
    • NOT NULL for required fields
    • UNIQUE for alternate keys (e.g., username, email)
    • CHECK constraints for simple domain rules (e.g., Age >= 0 AND Age <= 150)
    • Default values for common fallbacks (e.g., CreatedAt DEFAULT NOW())
  • The designer supports all of these directly in the row editor.

5. Index Strategically

  • Add indexes on:
    • Foreign key columns (improves joins)
    • Columns used in WHERE, ORDER BY, or GROUP BY clauses
    • Frequently searched text fields (consider GIN for JSONB or full-text)
  • Avoid over-indexing — each index adds write overhead.
  • Use the Add Index option to create composite or partial indexes when needed.

6. Plan Relationships Early

  • Design one-to-many, many-to-many, and one-to-one relationships using reference columns.
  • Set appropriate On Delete / On Update actions (e.g., CASCADE, RESTRICT, SET NULL).
  • Visualize relationships in the designer before building — broken references will block export.

7. Document Everything

  • Fill in Table Description and Column Description fields.
  • Exported Markdown documentation pulls directly from these — good habits here save time later.
  • When creating templates, add rich overviews and per-table descriptions for future users.

8. Leverage Secure Media Features

  • For file uploads or sensitive binary data:
    • Use the built-in SecureMedia and SecureMediaSession types.
    • Mark columns as Media or Encrypted (Non-Media) as appropriate.
  • This ensures encryption at rest and controlled access patterns. It is however a custom type, so if you know what you're doing you can select something like bytes and set it to an array.

9. Validate Often

  • Run Build Check frequently during design.
  • Fix all validation errors before exporting — this prevents runtime surprises.
  • Test generated SQL in a local PostgreSQL instance if possible.

10. Use Templates for Reuse

  • Create Row Templates for common patterns (e.g., Audit fields, Soft-delete, Multilingual).
  • Create Project Templates for full application blueprints (e.g., Blog CMS, E-commerce core).
  • Version your templates — the tool auto-increments v1, v2, etc.

11. Keep Performance in Mind

  • Use appropriate data types (e.g., INTEGER instead of BIGINT unless needed).
  • Prefer fixed-length types when possible (e.g., CHAR(2) for country codes).
  • For large text, consider TEXT over VARCHAR without limit.
  • Enable array types only when truly needed — they can complicate queries.

12. Plan for Evolution

  • Design with future changes in mind:
    • Add new columns as nullable first
    • Use soft-delete (IsDeleted BOOLEAN DEFAULT FALSE) instead of hard deletes
    • Version configuration tables rather than altering core schema

13.Most Importantly...

  • Use a paper and pencil to draft what you want to do before you do it. Using PsuedoCode to draft out a template of what you want to have working and figuring out the limitations + things to set for future updates (Even if you don't actively use them yet) is always a SSS+

By following these practices, your databases will be cleaner, safer, faster, and much easier to maintain as your application grows.
